## 1. デザインシステム

### 1.1 デザインシステムの目的

Nagare の画面は、AI を実行するためのランチャーではない。ユーザーが自分のドメインで品質の高い AI 利用を行うために、レビュー観点を品質基準として定義し、AI、モデル、スキル、MCP を簡単に組み込み、成果物がレビュー観点を満たしているかを見ながら作業できる画面である。

このデザインシステムは、次の判断を一貫して支える。

- どのレビュー観点を品質基準にするか。
- どの AI、モデル、スキル、MCP を作業に使うか。
- どの観点が満たされ、どれが未確認か。
- 人間がいま確認すべきことは何か。
- 採用、差し戻し、追加確認、引き継ぎのどれを選ぶべきか。

## 2. ブランド・プロダクトトーン

### 2.1 Hachiware Labs から引き継ぐ人格

Hachiware Labs の表現から、Nagare は次の人格を引き継ぐ。

- 人と AI が自然に協働する。
- 小さく試し、現場で使い、学びを改善へ回す。
- 人の思考を止めない。
- AI を過信せず、人間が判断できる道具にする。
- 実験的だが、画面は落ち着いていて実務に耐える。

Nagare では、この人格を「静かな作業台」として表現する。派手な未来感、過度なグラデーション、マーケティング的な hero 表現は使わない。

### 2.2 トーン

| 項目 | 方針 |
| --- | --- |
| 体験の印象 | 静か、明確、実務的、慎重、信頼できる |
| コピー | 短く、判断に必要な意味を残す |
| 視覚 | 白地、slate 系テキスト、indigo 系アクセント、控えめな境界線 |
| 密度 | ダッシュボードより作業台。情報は多いが、判断単位で整理する |
| 動き | 装飾ではなく、状態、因果、注意、完了を伝えるために使う |

## 3. デザイン原則

### 3.1 問いの順番を配置する

各画面は、情報を並べるのではなく、ユーザーが知りたい問いの順番で構成する。

1. これは何の画面か。
2. 自分が今見るべき状態か。
3. 何が重要か。
4. なぜそう言えるか。
5. どのレビュー観点が満たされたか。
6. どこを詳しく見ればよいか。
7. 次に何をすればよいか。
8. その操作は安全か。

### 3.2 品質はレビュー観点で表す

「高品質」「成功」「完了」は単独で使わない。必ず、どのレビュー観点が満たされたか、未確認か、差し戻されたかで表す。

良い例:

- `5/7 観点を確認済み。セキュリティ観点は人間レビュー待ち。`
- `採用不可: 完了条件は満たすが、根拠リンクが不足。`

避ける例:

- `AI が完了しました。`
- `品質は良好です。`

### 3.3 簡単さは構成の隠蔽ではなく、最小入力で表す

Nagare は AI、モデル、スキル、MCP の存在を隠さない。ただし、初回に全設定を要求しない。

- 作りたいものが決まっていない場合は、ビルドインの汎用品質基準で始める。
- ドメイン、成果物の種類、レビュー観点は候補として出し、合わない場合だけ修正する。
- 触ってよい範囲は依頼作成時または開始前確認で扱い、初回設定では要求しない。
- AI 構成の詳細は、必要になったときに提案として出す。
- 接続状態、権限、制約は、作業開始前の確認にまとめる。

### 3.4 人間の要対応を前面に出す

画面上で最も強く見せるのは、AI の内部ログではなく、人間が判断すべきこと。

- レビュー待ち。
- 採用可否。
- 差し戻し理由。
- 追加確認。
- 権限確認。
- 引き継ぎ判断。

### 3.5 詳細は掘れるが、主判断を邪魔しない

raw log、transcript、JSON、モデルパラメータ、API 名、内部イベントは初期表示に出さない。必要時に drawer、inspector、details disclosure で開く。

## 4. カラーシステム

### 4.1 基本方針

色は装飾ではなく、状態と判断のために使う。Hachiware Labs の白地、slate、indigo の落ち着いた方向を受け継ぎつつ、Nagare ではレビュー状態を読み分けるための状態色を明確に持つ。

### 4.2 セマンティックカラー

| 用途 | Token | 値 | 使い方 |
| --- | --- | --- | --- |
| 背景 | `--color-bg` | `#ffffff` | アプリ全体の基本背景 |
| 弱背景 | `--color-bg-subtle` | `#f8fafc` | サイドバー、ツールバー、薄い区切り |
| 面 | `--color-surface` | `#ffffff` | パネル、リスト行、フォーム |
| 面 hover | `--color-surface-hover` | `#f1f5f9` | リスト行 hover、選択前の反応 |
| 境界 | `--color-border` | `#e2e8f0` | 通常の境界線 |
| 強境界 | `--color-border-strong` | `#cbd5e1` | 選択中、重要な区切り |
| 本文 | `--color-text` | `#0f172a` | 主テキスト |
| 副本文 | `--color-text-muted` | `#475569` | 説明、補足 |
| 弱本文 | `--color-text-subtle` | `#64748b` | メタ情報、時刻 |
| Primary | `--color-primary` | `#4f46e5` | 主操作、選択、focus |
| Primary hover | `--color-primary-hover` | `#4338ca` | 主操作 hover |
| Primary weak | `--color-primary-subtle` | `#eef2ff` | 選択中背景、軽い強調 |
| Success | `--color-success` | `#16a34a` | 確認済み、採用可能 |
| Success weak | `--color-success-subtle` | `#dcfce7` | 確認済み badge 背景 |
| Warning | `--color-warning` | `#d97706` | 未確認、注意、レビュー待ち |
| Warning weak | `--color-warning-subtle` | `#fef3c7` | 注意 badge 背景 |
| Danger | `--color-danger` | `#dc2626` | 採用不可、失敗、危険操作 |
| Danger weak | `--color-danger-subtle` | `#fee2e2` | 失敗 badge 背景 |
| Info | `--color-info` | `#0284c7` | 補足、処理中、参考情報 |
| Info weak | `--color-info-subtle` | `#e0f2fe` | 情報 badge 背景 |

### 4.3 色の使用制限

- Primary は CTA、選択状態、focus、主要な接続線に限定する。
- Success は「レビュー観点を満たした」場合にだけ使う。
- Warning は「未確認」「人間レビュー待ち」「根拠不足」に使う。
- Danger は「採用不可」「危険な権限」「失敗」に限定する。
- 色だけで状態を伝えない。必ずアイコン、ラベル、説明を併用する。

## 5. タイポグラフィ

### 5.1 フォント

| 用途 | フォント |
| --- | --- |
| UI 全体 | system-ui, -apple-system, BlinkMacSystemFont, "Segoe UI", "Noto Sans JP", sans-serif |
| コード、ログ、識別子 | "SFMono-Regular", Consolas, "Liberation Mono", monospace |

日本語 UI では読み取りやすさを優先する。本文、ラベル、ボタンで letter spacing は 0 を基本とする。ブランド表記や小さな eyebrow のみに限定して字間を広げてもよい。

### 5.2 サイズスケール

| Token | Size | Line height | 用途 |
| --- | ---: | ---: | --- |
| `text-xs` | 12px | 16px | badge、meta、時刻 |
| `text-sm` | 14px | 20px | ラベル、表、補足 |
| `text-base` | 16px | 24px | 入力、本文 |
| `text-lg` | 18px | 28px | パネル見出し |
| `text-xl` | 20px | 30px | 画面内主見出し |
| `text-2xl` | 24px | 32px | ページタイトル |
| `text-3xl` | 30px | 38px | 空状態や開始案内の見出し |

### 5.3 文字階層

- 画面タイトルは短くする。
- 判断対象の名前、現在状態、レビュー観点の結果を見出し相当に扱う。
- 長い説明は本文にしない。短い要約と details disclosure に分ける。

## 6. レイアウトシステム

### 6.1 基本構造

Nagare は作業アプリであるため、基本は app shell とする。

| 領域 | 役割 |
| --- | --- |
| 左ナビ | 主要導線、ドメイン、作業、品質記録 |
| 上部バー | 現在文脈、接続状態、ユーザー操作 |
| 中央 | 主判断領域 |
| 右 inspector | 根拠、レビュー観点、構成理由、詳細 |
| 下部 action bar | 採用、差し戻し、追加確認、開始などの判断後操作 |

### 6.2 推奨レイアウト

- Desktop: `240px nav / 1fr main / 360px inspector`
- Tablet: nav は collapsed、inspector は drawer
- Mobile: 1 column、要対応と CTA を上位に置く

### 6.3 Z配置

| 位置 | Nagare で置くもの |
| --- | --- |
| 左上 | 画面名、対象ドメイン、作業名 |
| 右上 | 接続状態、権限、AI構成、最終更新 |
| 中央左 | いま見るべき状態、レビュー結果、未確認事項 |
| 中央右 | 根拠、構成理由、品質記録、比較 |
| 左下 | 履歴、詳細、ログ、関連作業 |
| 右下 | 採用、差し戻し、追加確認、開始、引き継ぎ |

### 6.4 余白

| Token | Size | 用途 |
| --- | ---: | --- |
| `space-1` | 4px | アイコンとラベル |
| `space-2` | 8px | 小要素間 |
| `space-3` | 12px | フォーム行内 |
| `space-4` | 16px | パネル内 |
| `space-6` | 24px | セクション間 |
| `space-8` | 32px | 画面内大区切り |

余白の使い方:

- 主要判断領域の内側 padding は desktop で 24px、tablet で 20px、mobile で 16px を基本にする。
- リスト、テーブル、チェックリストの行間は 12px から 16px に抑え、判断単位を崩さない。
- 画面内で 32px を超える余白は、空状態、危険確認、レビュー完了など、状態の切り替わりを明確にする場面に限定する。
- 右 inspector は情報を詰めるのではなく、`要約 -> 根拠 -> 詳細` の順に 16px 区切りで積む。
- CTA 周辺は 16px 以上の余白を確保し、隣接する補助操作や danger 操作と誤タップさせない。

### 6.5 密度

Nagare は作業台型 UI とする。情報量は多いが、同時に読ませる情報は制限する。

| 領域 | 密度ルール |
| --- | --- |
| 作業キュー | 1行 56-64px。作業名、状態、レビュー進捗、最終更新だけを常時表示する。 |
| Review Checklist | 1観点 64-88px。観点名、状態、根拠有無、次アクションを1単位にする。 |
| AI構成カード | 1カード 96-128px。構成名、用途、権限、品質記録の要約だけを見せる。 |
| Timeline | イベント間 16px。現在工程と人間待ち以外は抑制表示にする。 |
| Inspector | 360px 幅を基準にし、1画面内で 3 ブロック以上を同時に強調しない。 |

密度の制限:

- 1画面の primary CTA は原則 1 つ。判断が分岐するレビュー画面だけ `採用 / 差し戻し / 追加確認` を並べる。
- 初期表示で同時に見せる状態 badge は、主要領域ごとに最大 3 種までにする。
- テーブルは横スクロール前提にしない。列が増える場合は inspector に逃がす。
- 1つのカードに、説明文、メトリクス、ログ、CTA を全部載せない。

### 6.6 固定寸法とレスポンシブ

| 要素 | Desktop | Tablet | Mobile |
| --- | --- | --- | --- |
| 左ナビ | 240px 固定 | 64px collapsed | 下部 navigation または drawer |
| 右 inspector | 360px 固定 | drawer | full-screen drawer |
| 下部 action bar | 72px | 72px | 88px、主要CTAを下固定 |
| Review Checklist 行 | min 64px | min 64px | min 72px |
| Status chip | min 28px 高 | min 28px 高 | min 32px 高 |

テキスト量で高さが大きく変わる領域は、2行まで表示し、詳細は disclosure で開く。ボタン、badge、行高は状態変化で伸縮させない。

### 6.7 情報構造

各画面の情報は、次の5層に分ける。

1. 現在文脈: 画面名、対象ドメイン、作業名、状態。
2. 主判断: 人間が今判断すること。
3. 判断材料: レビュー観点、根拠、影響範囲、構成理由。
4. 次行動: 採用、差し戻し、追加確認、開始、引き継ぎ。
5. 詳細: raw log、transcript、内部パラメータ、全履歴。

初期表示で前面に出すのは 1 から 4 まで。5 は inspector、drawer、details disclosure に置く。画面内の見出しはこの構造に沿わせ、システム内部の分類をそのままナビゲーションにしない。

## 7. 形状・境界・影

### 7.1 Radius

| Token | Size | 用途 |
| --- | ---: | --- |
| `radius-sm` | 4px | badge、入力内チップ |
| `radius-md` | 8px | ボタン、入力、パネル、カード |
| `radius-lg` | 12px | modal、drawer、空状態パネル |

カードやパネルは原則 8px。過度に丸い UI は使わない。

### 7.2 Border

- 通常境界は 1px `--color-border`。
- 選択中は 1px `--color-primary` または左 border 3px。
- 危険状態は border と背景を併用し、色だけに依存しない。

### 7.3 Shadow

影は階層表現に限定する。

| Token | 用途 |
| --- | --- |
| `shadow-sm` | hover、軽い浮き上がり |
| `shadow-md` | dropdown、popover |
| `shadow-lg` | modal、drawer |

通常のパネルは影ではなく border で区切る。

## 8. コンポーネント原則

### 8.1 Button

| 種類 | 用途 |
| --- | --- |
| Primary | 作業開始、採用、保存など主操作 |
| Secondary | 追加確認、詳細確認、下書き保存 |
| Tertiary | キャンセル、戻る、補助操作 |
| Danger | 危険な差し戻し、権限解除、破棄 |

ルール:

- CTA は判断材料を見た後に置く。
- 採用と差し戻しは近くに置くが、色と文言を明確に分ける。
- 危険操作は確認 dialog または inline confirmation を挟む。

### 8.2 Badge / Status Chip

レビュー状態を即読できるようにする。

| 状態 | 表示 |
| --- | --- |
| 確認済み | `確認済み` success |
| 未確認 | `未確認` warning |
| レビュー待ち | `レビュー待ち` warning |
| 採用不可 | `採用不可` danger |
| 処理中 | `処理中` info |
| 引き継ぎ可能 | `引き継ぎ可` info |

### 8.3 Review Checklist

Nagare の中核コンポーネント。各レビュー観点は以下を持つ。

- 観点名。
- 期待状態。
- 現在結果。
- 根拠。
- 判定者。
- 未確認理由。
- 次の行動。

状態は `満たした / 未確認 / 不足 / 人間確認待ち / 対象外` の5つを基本にする。

### 8.4 AI Composition Card

AI、モデル、スキル、MCP の構成を1つの判断単位として表示する。

表示項目:

- 構成名。
- 担当工程。
- 含まれる AI、モデル、スキル、MCP。
- 必要権限。
- 品質記録。
- 推奨理由。
- 確認ゲート。

### 8.5 Evidence Panel

根拠は成果物の近くに置く。

- 要約を先に出す。
- 詳細、ログ、raw data は折りたたむ。
- 根拠が不足している場合は「不足」と明示する。

### 8.6 Timeline

作業の因果関係を示す。

基本イベント:

- 依頼作成。
- AI構成決定。
- 計画。
- 実装。
- 補助確認。
- レビュー。
- 差し戻し。
- 引き継ぎ。
- 採用。

現在地点と人間待ちは強調する。

### 8.7 Inspector / Drawer

主判断に不要な詳細は inspector または drawer に逃がす。

使う情報:

- raw log。
- transcript。
- 内部イベント。
- 接続詳細。
- モデルパラメータ。
- MCP request / response。

### 8.8 Input Composer

依頼作成や品質設定では、自由入力欄を増やさない。ユーザーが書くべきものは「判断に必要な文脈」に限定し、構造化できる部分は選択、提案、既定値で補う。

入力の優先順位:

1. ユーザーにしか分からないこと: 依頼内容、ドメイン、触ってよい範囲、採用判断。
2. ユーザーが確認すべきこと: レビュー観点、権限、危険範囲、差し戻し理由。
3. システムが提案できること: 完了条件、AI構成候補、レビューゲート、改善候補。
4. 初期表示に出さないこと: モデルパラメータ、詳細プロンプト、内部ログ、API request。

ルール:

- 必須入力は 1 画面あたり 3 項目を目安にする。4 項目以上になる場合は、既定値または次画面確認へ分離する。
- 自由入力より、選択済みレビュー観点、過去の依頼、ドメイン品質基準からの補完を優先する。
- 任意入力は初期表示で畳み、`条件を追加`、`詳細を指定` のような disclosure に置く。
- 差し戻し理由のように品質記録へ残る入力は必須にするが、未達レビュー観点から文案を下書きする。

### 8.9 Built-in Policy

Nagare は、依頼内容から自動推定できるものを逐一ユーザーに確認させない。通常時はビルトインポリシーで自動適用し、判断理由を監査記録に残す。ユーザー確認は、リスク、曖昧さ、信頼度不足がある場合だけ求める。

基本方針:

- 通常時は自動適用する。
- 怪しい時だけ確認する。
- すべての自動判断は記録に残す。
- ユーザーは後から判断理由を確認、差し戻し、改善できる。

自動適用するもの:

- ビルドイン汎用品質基準。
- 推定ドメイン。
- 推定された触ってよい範囲。
- 完了条件候補。
- レビュー観点候補。
- 既定AI構成。
- 依頼内容に合うスキル、MCP 候補。

確認を求める条件:

| 条件 | 確認内容 |
| --- | --- |
| 書き込み、削除、外部送信を含む | 対象範囲と権限 |
| 触ってよい範囲が広い | 範囲分割または明示許可 |
| 推定ドメインの信頼度が低い | ドメイン候補の選択 |
| 完了条件が曖昧 | 完了条件候補の採用または修正 |
| レビュー観点が不足 | 追加確認または汎用品質で続行 |
| AI構成が未接続 | 接続または別構成選択 |
| 危険権限を持つスキル/MCP が必要 | スキル/MCP の認可 |
| 過去に同種作業で失敗が多い | レビューゲート追加 |

監査記録に残すもの:

- 自動推定したドメイン、範囲、完了条件。
- 適用した品質基準とレビュー観点。
- 選定した AI 構成、スキル、MCP。
- 自動適用した理由。
- ユーザー確認を省略した理由。
- ユーザー確認を要求した理由。
- 作業開始前に表示した警告。

## 9. 状態設計

### 9.1 必須状態

各主要画面は以下を持つ。

- 初期状態。
- 空状態。
- 入力中。
- 接続確認中。
- 実行中。
- レビュー待ち。
- 未確認あり。
- 採用可能。
- 採用不可。
- 差し戻し後。
- 失敗。
- 引き継ぎ候補あり。
- 完了。

### 9.2 誤判断防止

- AI 推奨には必ず理由を添える。
- 未確認のレビュー観点がある場合、採用ボタンは弱めるか確認を挟む。
- 危険な権限を含む AI 構成は作業開始前に明示する。
- 差し戻し理由は空欄にできない。
- 行動後は結果を表示する。

### 9.3 状態表現の型

状態は `色 + アイコン + ラベル + 理由 + 次行動` の組で表す。色や badge だけで判断させない。

| 状態 | 色 | Icon | 表示位置 | 理由の出し方 | CTA |
| --- | --- | --- | --- | --- | --- |
| 初期状態 | neutral | `CircleHelp` | 主領域中央 | 何をすれば始められるかを3項目以内で表示 | 開始案内 |
| 入力不足 | warning | `AlertTriangle` | 入力欄直下、action bar | 不足項目を1行で表示 | 無効または下書き保存 |
| 接続未確認 | warning | `PlugZap` | 右上、AI構成領域 | 未確認の構成と必要操作を表示 | 接続を確認 |
| 実行中 | info | `LoaderCircle` | header、timeline | 現在工程と次の人間待ちを表示 | 停止、詳細 |
| 人間待ち | warning | `UserRound` | 中央左、作業キュー先頭 | 何に答えるべきかを表示 | 質問に回答、レビューする |
| 未確認あり | warning | `CircleAlert` | Review Checklist | 未確認の観点と理由を表示 | 追加確認 |
| 採用可能 | success | `CheckCircle2` | Review Checklist、decision bar | 確認済み観点と残課題を表示 | 成果物を採用 |
| 採用不可 | danger | `XCircle` | Review Checklist、decision bar | 不足観点と根拠不足を表示 | 差し戻す |
| 失敗 | danger | `OctagonAlert` | header、timeline、作業キュー | 失敗工程と影響範囲を表示 | 失敗を確認 |
| 完了 | success | `BadgeCheck` | header、品質記録 | 採用済みレビュー観点を表示 | 記録を見る |

### 9.4 状態の視覚優先度

1. 人間待ち、採用不可、失敗。
2. 未確認あり、危険範囲、接続未確認。
3. 採用可能、完了。
4. 実行中、参考情報。

同じ画面に複数状態がある場合は、上位状態だけを primary 表現にする。下位状態は badge、secondary panel、inspector に下げる。

## 10. モーション・マイクロインタラクション

### 10.1 原則

動きは状態と因果を伝えるために使う。常時動く装飾は使わない。

| 動き | Duration | 用途 |
| --- | ---: | --- |
| hover / press | 80-120ms | 操作反応 |
| drawer open | 160-220ms | 詳細が右から出る関係 |
| status transition | 180-240ms | 未確認から確認済みへの変化 |
| toast | 180ms in / 120ms out | 保存、接続確認、差し戻し完了 |

### 10.2 注意誘導

- 新しい人間待ちは 1 回だけ淡く強調する。
- レビュー観点が未確認から不足に変わったときは、その行だけ強調する。
- prefers-reduced-motion を尊重する。

### 10.3 画面別モーション

| 場面 | 動き | 目的 |
| --- | --- | --- |
| 作業キューに新しい人間待ちが入る | 行背景を 600ms だけ淡く強調 | 今見るべきものを知らせる |
| Review Checklist の状態が変わる | 対象行の status chip を 180ms で切り替え | どの観点が変わったかを示す |
| Evidence Inspector を開く | 右から drawer が 180ms で開く | 成果物と根拠の空間関係を示す |
| 差し戻し理由を生成する | 下書きテキストを skeleton から表示 | AI が文案を補助していることを示す |
| 作業開始 | action bar を progress 状態へ 220ms で変える | 開始操作と実行状態の因果を示す |
| 失敗検知 | timeline の失敗工程を danger border で表示 | 失敗箇所と影響範囲に注意を向ける |
| 採用完了 | decision bar を success summary に置換 | 操作後の結果を画面上に残す |

### 10.4 使わない動き

- 常時揺れる badge。
- 背景の装飾アニメーション。
- 成果物レビュー中の自動スクロール。
- 複数箇所を同時に光らせる注意誘導。
- 重要な状態を toast だけで消す表現。

## 11. コピーライティング

### 11.1 基本

- 主語はユーザーの作業にする。
- 内部実装語を前面に出さない。
- AI を擬人化しすぎない。
- 「完了」ではなく「何が確認されたか」を書く。

### 11.2 用語

| 使う | 避ける |
| --- | --- |
| レビュー観点 | チェック項目だけ |
| 品質基準 | 高品質 |
| AI構成 | ツール選定 |
| 根拠 | ログだけ |
| 採用 | 承認だけ |
| 差し戻し | 失敗だけ |
| 追加確認 | 再実行だけ |

### 11.3 CTA 文言

| 状況 | CTA |
| --- | --- |
| 汎用品質で開始 | `汎用品質で始める` |
| 品質基準を調整 | `品質基準を保存` |
| 依頼作成 | `作業内容を確認` |
| AI構成確認 | `この構成で開始` |
| レビュー待ち | `レビューする` |
| 採用可能 | `成果物を採用` |
| 不足あり | `差し戻す` / `追加確認する` |
| 失敗 | `回復方法を選ぶ` |
| 引き継ぎ | `引き継ぎ内容を確認` |

## 12. アクセシビリティ

- 主要操作は keyboard で到達できる。
- focus ring は `--color-primary` で明示する。
- 状態は色だけでなく、テキストとアイコンで伝える。
- コントラストは WCAG AA を満たす。
- 表、一覧、timeline はスクリーンリーダーで順序が崩れないようにする。
- toast だけで重要結果を伝えない。画面上にも状態を残す。

## 13. 画面仕様への適用ルール

今後の各画面仕様は、必ず以下を持つ。

1. ユーザー。
2. 利用文脈。
3. ユーザーの問いの順番。
4. Z配置。
5. 主要コンポーネント。
6. 表示情報。
7. 初期表示で隠す情報。
8. 状態。
9. 誤判断防止。
10. 主要 CTA。

この順序で定義しない画面は、見た目が整っていても Nagare の画面仕様として扱わない。

### 14.5 汎用画面モデル

Nagare の画面は、ドメインが変わっても次の型で成立させる。

| 型 | 使う画面 | ユーザーの問い | 必要最小入力 |
| --- | --- | --- | --- |
| Connect | `SCR-00A` | AIを使える状態か | 接続済みAIがない場合だけ接続先と認証 |
| Define | `SCR-01` | 何を品質とみなすか | なし。ビルドイン汎用品質で始め、候補が合わない場合だけ修正 |
| Equip | `SCR-02` | どの AI 構成を使える状態にするか | なし。使いたい構成がない場合だけ追加 |
| Ask | `SCR-03` | 何を頼みたいか | 依頼内容 |
| Confirm | `SCR-04` | この内容で始めて安全か | 危険範囲がある場合の確認 |
| Watch | `SCR-00`, `SCR-05` | 今、人間が見るべきことは何か | 人間待ちがある場合の回答 |
| Review | `SCR-06` | 採用できる根拠があるか | 判断後の採用、差し戻し、追加確認 |
| Recover | `SCR-07` | どう続ければ品質を崩さないか | 判断後の回復方法 |
| Learn | `SCR-08` | 次回の品質基準へ何を反映するか | 判断後の改善候補採用 |

この型により、画面は機能名ではなくユーザーの判断で並ぶ。新しいドメインや AI 構成を追加しても、ユーザーは `Define -> Ask -> Review -> Learn` の基本線から外れない。

### 14.6 代表ユースケースストーリー

#### UC-01 ドメイン責任者が、AI利用の品質基準を確認する

状況:

- ユーザーは、法務、開発、リサーチ、営業資料など、特定ドメインの成果物品質に責任を持つ。
- AI を使いたいが、何をレビューすれば十分なのかが毎回曖昧になっている。
- モデルや MCP の細かい設定より、成果物が満たすべき観点を先に固めたい。

最小入力:

- なし。作りたいものが決まっていない場合は、ビルドイン汎用品質を適用する。
- 現在文脈、過去の作業、テンプレートから候補を表示する。
- 候補が合わない場合だけ、ドメイン名や成果物種類を修正する。

画面が支える判断:

- ビルドイン汎用品質のまま始めるか、提案されたレビュー観点を採用、削除、追加するか。
- 人間レビュー必須にする観点はどれか。
- この品質基準で依頼を始めてよいか。

成功状態:

- AI に渡る品質基準が短く確認できる。
- ユーザーは内部設定を見ずに、レビュー観点として品質を定義できる。
- 作りたいものが未定でも、汎用品質基準で最初の依頼へ進める。

#### UC-02 実務担当者が、最小入力でAIに依頼する

状況:

- ユーザーは、作りたい成果物や直したい対象は分かっている。
- ただし、どの AI、モデル、スキル、MCP を使うべきかは判断したくない。
- 依頼内容を長いプロンプトに整える作業も負担になっている。

最小入力:

- 何をしてほしいか。

画面が支える判断:

- 自動提案されたドメイン、範囲、完了条件が妥当か。
- 危険範囲や権限を許可してよいか。
- 高性能AIで計画し、適切なAI構成で実行する流れを使ってよいか。

成功状態:

- ユーザーは依頼内容だけで開始候補まで進める。
- 追加設定は、危険範囲、権限、レビュー観点の不足がある時だけ求められる。

#### UC-03 レビュー担当者が、成果物を根拠つきで採用判断する

状況:

- ユーザーは、AI が作った成果物をそのまま採用してよいか不安がある。
- 何となく良さそうではなく、レビュー観点ごとに根拠を見たい。
- 差し戻す場合は、次の AI 作業へ伝わる形で理由を残したい。

最小入力:

- なし。まず成果物、レビュー観点、根拠を読む。

画面が支える判断:

- 採用できる観点、未確認の観点、不足している観点は何か。
- 根拠不足なら追加確認にするか、差し戻すか。
- 差し戻し理由を品質記録として残してよいか。

成功状態:

- 採用、差し戻し、追加確認の判断が、レビュー観点と根拠に紐づく。
- 次回の品質基準や AI 構成の改善に記録が残る。

#### UC-04 チーム管理者が、品質記録からAI構成を改善する

状況:

- ユーザーは、複数の AI、モデル、スキル、MCP をチームで使っている。
- 費用を直接管理したいのではなく、品質記録をもとに結果として無駄な試行を減らしたい。
- 採用率だけでなく、どのレビュー観点で差し戻しが起きたかを見たい。

最小入力:

- なし。まず期間とドメインの既定表示を見る。

画面が支える判断:

- どのレビュー観点を強化すべきか。
- どの AI 構成を次回から候補にするか。
- 改善候補を品質基準、依頼テンプレート、AI構成のどこへ反映するか。

成功状態:

- 品質記録が次回の設定に返る。
- コストは管理対象ではなく、品質の安定と再試行削減の結果として下がる。

#### 18.4.1 Skill 追加の操作

入口:

- `Skills` 一覧ヘッダー右側の `Skillを追加` を押す。
- 依頼作成中に不足する Skill が推奨された場合は、候補から `Skillを追加` を押す。

基本フロー:

1. 追加元を選ぶ。
2. Nagare が `SKILL.md` または package metadata を読み、名前、説明、用途、バージョン、依存情報を抽出する。

4. `Skillを追加` を押す。
5. catalog に登録する。
6. すぐ使う場合は、追加後に該当 Agent 詳細の `Skills / MCP` へ遷移する。

Skill 追加時にユーザーが指定するもの:

| 項目 | 必須 | 指定方法 | 補足 |
| --- | --- | --- | --- |
| 追加元 | 必須 | 組み込み候補、GitHub URL、ローカルフォルダ、zip/package | `SKILL.md` を含むことを検証する |
| 用途 | 原則自動 | 調査、実装、レビュー、文章化、確認などから選択 | metadata から推定し、合わない場合だけ修正する |
| 適用ドメイン | 任意 | 汎用、開発、文章、調査など | 未指定なら汎用候補として扱う |
| バージョン固定 | 原則自動 | commit、tag、checksum | 更新で実行中作業へ影響しないよう固定する |

Skill 追加時にユーザーへ見せる確認:

- 読み取った Skill 名、説明、追加元。
- 現在使用しているAgentと、そのAgent詳細へのリンク。
- Agentに割り当てた場合、実行時に copy されること。
- 依存する MCP や外部ツールがある場合の不足。
- checksum、version、登録日時。

Skill 追加でユーザーに要求しないもの:

- catalog 内部パス。
- OS 別の保存先。
- materialization manifest の手入力。
- Skill ファイルのコピー先指定。

#### 18.4.2 MCP 追加の操作

入口:

- `MCP` 一覧ヘッダー右側の `MCPを追加` を押す。
- 依頼作成または作業開始前確認で、必要な MCP が不足している場合は `MCPを追加` を押す。

基本フロー:

1. 接続テンプレートまたは接続方式を選ぶ。
2. 接続に必要な最小項目だけ入力する。
3. 読み取り、書き込み、外部送信、ローカル実行の権限を確認する。
4. `接続テスト` を押す。
5. 成功したら `MCPを追加` を押す。
6. catalog に登録する。
7. すぐ使う場合は、追加後に該当 Agent 詳細の `Skills / MCP` へ遷移する。

MCP 追加時にユーザーが指定するもの:

| 項目 | 必須 | 指定方法 | 補足 |
| --- | --- | --- | --- |
| 接続方式 | 必須 | 組み込み、stdio command、HTTP/SSE、ローカルツール | 組み込みは入力を最小化する |
| 接続先 | 必須 | command、URL、対象フォルダ、対象アカウントなど | 方式により項目を出し分ける |
| 権限 | 必須 | read、write、external send、local exec の選択 | 危険権限は開始前確認にも出す |
| 対象範囲 | 必須 | フォルダ、リポジトリ、URL、DB、ワークスペースなど | 範囲なしの広域許可は避ける |
| 認証情報 | 必要時 | secret 名またはOS資格情報の参照 | secret 値は画面に表示しない |

MCP 追加時にユーザーへ見せる確認:

- 現在使用しているAgentと、そのAgent詳細へのリンク。
- 読み取り、書き込み、外部送信、ローカル実行の有無。
- 対象フォルダや対象URLなど、実際に触れる範囲。
- 接続テスト結果。
- 依頼作成時・作業開始前確認で追加確認が必要になる条件。

MCP 追加でユーザーに要求しないもの:

- 全エージェント共通の MCP 設定ファイル編集。
- OS 別のパス書式。
- secret 値の再表示。
- 未使用エージェント向けの設定。

### 18.7 状態

#### 誤判断防止

- 危険な権限を含む MCP は、用途と許可範囲を同じ場所に表示する。

- `この構成を有効化` は接続確認が通るまで無効にする。
- 接続や認可は、その構成を使う意図がある場合だけ要求する。

### 18.8 Skill / MCP のクロスプラットフォーム materialization

スキルと MCP は、Nagare 全体ではカタログとして管理する。ただし、実行時に全エージェントへ共通公開しない。各エージェントの allowlist に従い、実行フォルダへ必要なものだけ materialize する。

Windows / Mac / Linux 共通で同じ挙動にするため、標準方式は次の通りとする。

| 対象 | 標準方式 | 理由 |
| --- | --- | --- |
| Skill | 実行フォルダへ copy | symlink 権限差、パス解決差、監査の曖昧さを避ける |
| MCP | エージェント別 `mcp.json` を生成 | 権限、環境変数、対象フォルダを agent ごとに分ける |
| 証跡 | `materialization.manifest.json` を生成 | どの skill/MCP をどの agent に見せたか追跡する |
| パス | 内部表現は `/` 区切り、実行時に OS パスへ変換 | Windows / Mac / Linux の差を UI と記録に出さない |

標準の実行フォルダ構造:

```text
.nagare/
  catalog/
    skills/
      hachi-search/
  runs/
    run-001/
      agents/
        researcher/
          skills/
            hachi-search/
          mcp.json
          materialization.manifest.json
        implementer/
          skills/
          mcp.json
          materialization.manifest.json
```

materialization の手順:

1. Run を作成する。
2. Agent ごとの実行フォルダを作成する。
3. Agent の `allowedSkills` と `allowedMcp` を読む。
4. 許可された Skill だけを catalog から実行フォルダへ copy する。
5. Agent ごとの `mcp.json` を生成する。
6. `materialization.manifest.json` に source、version、checksum、targetAgent、permissions、materializedAt を記録する。
7. その実行フォルダだけを参照して Agent を起動する。

制約:

- symlink は標準方式にしない。開発者向け最適化として許可する場合も、UI と監査上は copy と同じ結果に見える必要がある。
- 共通 skill フォルダを Agent ランタイムへ直接見せない。
- Agent 実行フォルダ外への相対パス参照を許可しない。
- Windows の予約名、パス長、大小文字差、改行コード差で失敗しないよう、copy 時に正規化と検証を行う。
- materialized skill は実行単位のスナップショットとして扱い、catalog 更新の影響を実行中の Agent に反映しない。

### 18.9 Agent 画面での Skill / MCP 割り当て

Skill / MCP の割り当て編集は、Skills 画面や MCP 画面では行わない。Agent の能力を変える操作なので、Agent 詳細画面の `Skills / MCP` セクションで行う。

Agent 詳細で表示するもの:

- このAgentに割り当て済みの Skills。
- このAgentに割り当て済みの MCP。
- MCP ごとの Agent 別許可範囲: 読み取り、書き込み、外部送信、ローカル実行、対象フォルダ、対象URL。
- materialization 予定: 実行時に copy される Skill と、生成される `mcp.json`。
- 参照元 catalog へのリンク。

Agent 詳細で操作するもの:

- Skill を追加、削除、有効化、無効化する。
- MCP を追加、削除、有効化、無効化する。
- MCP の Agent 別許可範囲を調整する。
- Agent の役割に合わない Skill / MCP を外す。

MCP 候補の絞り込み:

- MCP は Agent に割り当てるが、実際に利用できるかはその Agent が使うランタイムの MCP 設定スコープで決まる。
- Agent 詳細の MCP 追加候補には、対象ランタイムがプロジェクト単位またはセッション単位で安全に渡せる MCP だけを表示する。
- ユーザー全体またはグローバル設定にしか反映できない MCP は、通常候補に表示しない。
- Nagare 側プロキシで Agent 単位に制御できる場合だけ、`Nagare管理経由` と明示して候補に表示する。
- 候補がゼロ件の場合は、無効な選択肢を並べず、対象ランタイムでは利用可能な MCP がない理由と、ランタイム変更またはMCP接続設定への導線を表示する。
- MCP 管理画面では登録、接続、認証、対応ランタイムを確認する。Agent への割り当て操作は Agent 詳細で行う。

当面の判定方法:

MCP の利用可否は、動的検出や各ツールの詳細 probe ではなく、Runtime 種別ごとの固定テーブルで判定する。テーブルは実装側の `RuntimeMcpCapability` として持ち、Agent の MCP 候補表示では `agent_assignable = true` の Runtime だけを対象にする。

| Runtime | MCPスコープ | AgentへのMCP付与 | 備考 |
| --- | --- | --- | --- |
| Codex | project | 可 | プロジェクト限定として扱う |
| Codex CLI | project | 可 | 実行時に対象Agent向けへ materialize する |
| OpenClaw | global_only | 不可 | project/session 単位の制御が確認できるまで候補外 |
| Claude Code | project / local / user | 要追加 | Runtime種別として実装した時点でテーブルへ追加する |
| OpenCode | project / global | 要追加 | Runtime種別として実装した時点でテーブルへ追加する |

Agent追加時に扱う項目:

| 項目 | 必須 | 指定方法 | 補足 |
| --- | --- | --- | --- |
| Agentテンプレート | 必須 | researcher、implementer、reviewer などから選択 | 役割、初期プロンプト、推奨Skill/MCPを自動提案する |
| Agent名 | 必須 | 自動生成、必要時だけ修正 | `research-agent` のような表示名と識別名 |
| 接続先 | 必須 | 接続済みAI/ローカルAIから選択 | 未接続の場合は AI / Models へ誘導する |
| モデル | 必須 | 接続先に応じた候補から選択 | 計画、実装、レビューなどの用途別候補を出す |
| ドメイン品質 | 任意 | ビルドイン汎用、または登録済みドメイン | 未指定ならビルドイン汎用を使う |
| システムプロンプト | 必須 | テンプレートとドメイン品質から自動生成、必要時だけ編集 | 空欄では保存できない |
| Skills | 任意 | 登録済みSkillから選択 | Agentの能力として割り当てる |
| MCP | 任意 | 登録済みMCPから選択 | Agent別の対象範囲と権限を設定する |

Agent追加時にユーザーへ見せる確認:

- このAgentが使う接続先とモデル。
- 適用するドメイン品質基準。
- 生成されたシステムプロンプトの要約と全文編集入口。
- 割り当てる Skill / MCP と、MCP の対象範囲。
- 実行時に materialize される Skill と生成される `mcp.json`。

Agent追加でユーザーに要求しないもの:

- raw model parameter。
- API key 値。
- materialization manifest の手入力。
- OS 別の実行フォルダ。

Skills / MCP 画面で表示するもの:

- 使用中Agentの一覧。
- 使用中Agentへのリンク。
- 使用中の場合の削除影響。
- 常設の右側Inspectorは置かない。一覧の使用中Agent表示と重複するため、詳細は `編集` または `削除` 操作時にだけ drawer / modal で表示する。

Skills / MCP 画面で操作するもの:

- catalog への追加。
- 登録物そのものの編集。
- 未使用または影響確認済み登録物の削除。
- 接続テスト、バージョン固定、checksum確認。

削除時の誤判断防止:

- 使用中の Skill / MCP を削除する場合、影響する Agent と実行予定の作業を表示する。
- 実行中 run の materialized snapshot は削除で壊さない。
- 使用中のものは即時削除ではなく、`新規runでは使用不可`、`既存runは維持` を明示する。
- secret 値は削除確認でも表示しない。

- 作業停止。
- 追加確認の依頼。

### 21.6 初期表示で隠すもの

### 21.7 状態

## 24. SCR-08 品質記録・改善

### 24.1 目的

レビュー、差し戻し、失敗、採用結果を次回の品質基準、依頼文、AI構成に活かせるようにする。

Quality Records は、個別作業のログ置き場ではない。ドメインごとのルーブリックが妥当か、Agent ごとの振る舞いやプロンプトが期待どおりかを見つけ、修正先へ移動するための分析画面である。

### 24.2 ユーザーの問い

1. どのドメインで品質が安定しているか。
2. どのレビュー観点で差し戻しが多いか。
3. どの Agent で失敗、差し戻し、未確認が多いか。
4. どの AI、モデル、スキル、MCP の組み合わせが有効だったか。
5. 問題はドメインのルーブリック、Agentプロンプト、Skill/MCP、依頼テンプレートのどこにありそうか。
6. 次回、品質基準、依頼文、AI構成をどう変えるべきか。
7. チームで再利用できる設定は何か。

#### Z配置

| 位置 | 配置する情報 |
| --- | --- |
| 左上 | `品質記録・改善`、対象期間、対象ドメイン |
| 右上 | フィルタ、共有状態、更新日時 |
| 中央左 | Domain view: ドメイン別品質サマリー、差し戻しが多いレビュー観点 |
| 中央右 | Agent view: Agent別の成功率、差し戻し率、未確認率、失敗傾向 |
| 左下 | 最近の差し戻し理由、関連作業、該当レビュー観点 |
| 右下 | 改善候補、反映先、`ルーブリックを見直す`、`Agentプロンプトを見直す` |

### 24.3 表示するもの

- ドメイン別品質サマリー。
- レビュー観点別の確認率、差し戻し率。
- Agent別品質サマリー: 採用率、差し戻し率、失敗率、未確認率、平均再試行数。
- Agent別の問題観点: どのレビュー観点で失敗しやすいか、どの作業種別で失敗しやすいか。
- AI構成別の採用率、失敗傾向。
- 最近の差し戻し理由。
- 改善候補: ドメインルーブリック修正、Agentプロンプト修正、依頼テンプレート修正、AI構成変更、Skill/MCP見直し。
- チーム共有候補。

#### 主要コンポーネント

- Quality Summary Table。
- Domain Quality Matrix。
- Agent Quality Matrix。
- Review Trend Chart。
- AI Composition Performance Table。
- Improvement Suggestion List。
- Change Preview。
- Team Share Panel。

### 24.4 入力・選択させるもの

初期必須:

- なし。まず品質記録と改善候補を読む。

自動提案:

- 期間。
- ドメイン。
- Agent。
- 成果物種類。
- AI構成。
- 反映先。

必要時:

- 改善候補の採用可否。
- ドメイン品質基準への反映。
- ドメインルーブリックの修正。
- Agentプロンプトへの反映。
- Agentに割り当てる Skill / MCP の見直し。
- 依頼テンプレートへの反映。
- チーム共有範囲。

### 24.5 主CTA

- `品質基準に反映`
- `ルーブリックを見直す`
- `Agentプロンプトを見直す`
- `依頼テンプレートを更新`
- `AI構成を見直す`

### 24.6 初期表示で隠すもの

- 個別ログ。
- 全レビューコメント。
- 内部メトリクス。

### 24.7 状態

- 記録なし。
- 改善候補あり。
- 品質安定。
- 差し戻し増加。
- ドメインルーブリック見直し推奨。
- Agentプロンプト見直し推奨。
- 構成見直し推奨。

#### 誤判断防止

- 改善候補は自動反映せず、変更差分を表示してから適用する。
- サンプル数が少ない品質記録は `参考` と明示する。
- AI構成の採用率だけで推奨せず、差し戻し観点と根拠も併記する。
- Agent単体の成績だけでプロンプト修正を推奨せず、作業種別、ドメイン、レビュー観点、使用Skill/MCPを併記する。
- ドメインルーブリック修正を提案する場合、どのレビュー観点が曖昧、不足、過剰だったかを示す。
- Agentプロンプト修正を提案する場合、失敗した行動、期待される行動、プロンプト差分案を分けて表示する。
- チーム共有時は、対象ドメインと適用範囲を確認させる。

### 24.8 Domain / Agent View

Quality Records は、少なくとも次の2つの分析 view を持つ。

Domain view:

- ドメインごとの採用率、差し戻し率、失敗率。
- レビュー観点ごとの確認率、未確認率、差し戻し率。
- 差し戻し理由の多い観点。
- ルーブリックが曖昧、不足、過剰になっている可能性。
- `ルーブリックを見直す` から SCR-01 ドメイン品質設定へ遷移する。

Agent view:

- Agent ごとの採用率、差し戻し率、失敗率、未確認率。
- Agent が失敗しやすいドメイン、作業種別、レビュー観点。
- Agent が使った AI、モデル、Skill、MCP。
- Agentプロンプト、割り当てSkill/MCP、MCP権限の見直し候補。
- `Agentプロンプトを見直す` から Agent 詳細へ遷移する。

反映先の切り分け:

| 見つかった問題 | 主な反映先 | 例 |
| --- | --- | --- |
| 同じドメインで複数Agentが同じ観点を外す | ドメインルーブリック | レビュー観点の追加、表現の明確化 |
| 特定Agentだけが同じ観点を外す | Agentプロンプト | 注意事項、手順、出力形式の修正 |
| 特定Skill/MCP使用時だけ失敗が増える | AgentのSkill/MCP割り当て | Skillを外す、MCP権限を狭める |
| 依頼文の曖昧さが原因で失敗する | 依頼テンプレート | 必須入力や自動補完の改善 |
| 高性能AIの計画では成功し、軽量AIの実行で失敗する | AI構成 | 計画/実装/レビューの役割分担を変更 |

## 25. 共通 Drawer / Inspector

### 25.1 Evidence Inspector

根拠の詳細を確認するための共通 inspector。

表示:

- 根拠要約。
- 根拠の種類。
- 関連レビュー観点。
- 出典、差分、実行結果。
- 信頼性、未確認理由。

入力:

- 根拠を採用。
- 根拠不足として差し戻し。
- 追加確認を依頼。

誤判断防止:

- 根拠がない、または弱い項目は `採用` を primary CTA にしない。
- 根拠の出典、対象レビュー観点、確認日時を同じ領域に表示する。
- AI の主張と検証結果が一致しない場合は、検証結果を優先して表示する。
- 根拠不足のまま完了へ進む場合は、不足しているレビュー観点を明示する。

### 25.2 Raw Log Inspector

通常画面では見せない詳細ログを確認するための inspector。

表示:

- raw log。
- transcript。
- MCP request / response。
- モデル応答。
- 内部イベント。

入力:

- コピー。
- ダウンロード。
- 失敗診断へ送る。

誤判断防止:

- raw log は通常判断の主情報にしない。
- raw log 上の内部名、ID、モデル応答だけで品質判断させない。
- 失敗診断へ送る場合は、関連作業、工程、レビュー観点を自動で添える。
- 個人情報、シークレット、接続情報は masked 表示を標準にする。

### 25.3 AI構成 Inspector

作業に使われる構成の詳細を見る inspector。

表示:

- AI、モデル、スキル、MCP の一覧。
- 担当工程。
- 権限。
- 接続状態。
- 過去の品質記録。
- 選定理由。

入力:

- 構成を差し替える。
- 権限を確認する。
- 構成ライブラリで編集する。

誤判断防止:

- 構成の良し悪しを価格や速度だけで見せない。
- 過去の品質記録が少ない構成は `参考値` と明示する。
- 実行中作業の構成差し替えは、影響範囲と引き継ぎ方法を確認してから行う。
- MCP 権限の追加は、読み取り、書き込み、外部送信の違いを分けて確認させる。

## 26. ルーブリック観点での改善確認

この仕様書では、`ai_ui_design_rubric_v3.md` の評価で上限になりやすい点を、画面仕様に直接反映する。

| 観点 | 仕様書での対応 |
| --- | --- |
| ユーザー目的・ユースケース | 各画面に `目的` と `ユーザーの問い` を置き、代表ユースケースで利用文脈、不安、判断、成功状態を定義する。 |
| 問いの順番・Z配置 | `SCR-00` から `SCR-08` まで、画面ごとに Z 配置を明記する。 |
| 意図整合性 | デザイン原則、色、コピー、CTA を `品質観点を持ってAIを使う` 目的に寄せる。 |
| 情報構造・認知負担 | `現在文脈 -> 主判断 -> 判断材料 -> 次行動 -> 詳細` の5層に分け、raw log や内部設定を後段へ下げる。 |
| ユーザー入力の最小化 | `最小入力マトリクス` で、初期必須、自動提案、必要時入力を画面ごとに分け、閲覧後の判断を初期必須にしない。 |
| 余白・密度 | `余白`, `密度`, `固定寸法` を定義し、画面別の密度調整を指定する。 |
| モーション・マイクロインタラクション | 状態変化、保存、差し戻し、レビュー進行に限定し、画面別に使う動きを指定する。 |
| トーンマナー・視覚品質 | hachiware-labs.com の静かで構造的な印象を参照し、装飾より判断しやすさを優先する。 |
| 状態設計・誤判断防止 | 状態表現を `色 + アイコン + ラベル + 理由 + 次行動` に統一し、各画面と共通 inspector に `誤判断防止` を定義する。 |

90点を狙う採点目標:

| 観点 | 目標点 | 到達条件 |
| --- | ---: | --- |
| A. ユーザー目的・ユースケース | 14 / 15 | 代表ユースケースが画面構造と入力最小化に接続している。 |
| B. 問いの順番・Z配置 | 18 / 20 | 全画面で `状況 -> 重要点 -> 根拠 -> 詳細 -> 行動` が崩れない。 |
| C. 意図整合性 | 18 / 20 | 品質観点、AI構成、レビュー、記録が同じ目的に向いている。 |
| D. 情報構造・認知負担 | 14 / 15 | 初期表示は主判断に集中し、詳細は inspector に退避する。 |
| E. モーション・マイクロインタラクション | 9 / 10 | 状態、因果、注意、完了を支える動きだけを使う。 |
| F. トーンマナー・視覚品質 | 8 / 10 | 静かな作業台として一貫し、凡庸なSaaS画面に寄りすぎない。 |
| G. 状態設計・誤判断防止 | 9 / 10 | 根拠、未確認、危険範囲、操作後結果が常に見える。 |
| **合計** | **90 / 100** | 画面プロトタイプで余白、密度、状態、動きを実装確認する。 |

現時点の仕様到達ライン:

- 画面仕様書としては、65 点上限の主因だった `Z配置不足` を解消する。
- 情報設計は、機能一覧ではなくユーザーの判断順に構造化する。
- ユーザー入力は、ユーザーにしか分からない情報だけを初期必須にし、閲覧後の判断は初期必須にしない。
- 画面プロトタイプ実装時は、実際の visual density、余白、状態表現、モーション確認まで行い、90点前後を狙う。
