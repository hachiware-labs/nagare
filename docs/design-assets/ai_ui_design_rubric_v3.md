# AI生成UI デザイン評価ルーブリック v3

## 位置づけ

このガイドは、専門デザイナーの代替ではない。
AIが生成したWebサイト、アプリ画面、ダッシュボード、プロトタイプを、見た目だけで評価しないためのレビュー基準である。

特に以下を重視する。

- ユーザーの目的が具体化されているか
- ユーザーが順番に知りたくなる「問い」が配置されているか
- トーン、コピー、視線誘導、インタラクション、余白、色、CTAが同じ意図を向いているか
- 動きが装飾ではなく、理解・状態・注意・行動を支えているか
- 認知負担を下げるために、何を見せないかが設計されているか
- 生成物が「一般的な今風UI」ではなく、そのプロダクト固有の価値を表現しているか

---

## 中核原則

### 1. 良いUIは、情報ではなく「ユーザーの問いの順番」を配置する

UIは情報の陳列棚ではない。
ユーザーが目的へ向かう途中で、自然に抱く問いに順番に答える装置である。

例:

1. これは何の画面か
2. 自分に関係あるのか
3. 今、何が重要なのか
4. なぜそう言えるのか
5. どこを詳しく見ればよいのか
6. 次に何をすればよいのか
7. その行動は安全か
8. 行動後に何が起きたか

この問いの順番が設計されていないUIは、見た目が良くても高評価にしない。

---

### 2. デザインとは、意図の一貫性を視覚・言葉・動き・構造に分配する作業である

良いデザインは、個々の部品が美しいことではなく、すべての要素が同じ目的を支えていることで成立する。

評価対象:

- トーンマナー
- 情報構造
- 視線誘導
- コピー
- CTA
- 余白
- 色
- タイポグラフィ
- 状態表現
- マイクロインタラクション
- モーション
- 詳細情報の隠し方
- エラー、警告、確認、完了の表現

---

### 3. 「今風」は抽象語で評価しない

「今風」「洗練されている」「高級感がある」だけでは評価しない。
必ず、どの要素が何のために使われているかまで分解する。

例:

| 表現 | 分解して見るべきこと |
|---|---|
| 余白がある | 情報の優先順位を支えているか |
| グラデーションがある | ブランド感か、状態表現か、単なる装飾か |
| 動きがある | 状態・因果・注意・空間関係を伝えているか |
| カードが美しい | 判断単位としてまとまっているか |
| CTAが目立つ | 納得後の自然な行動位置にあるか |
| コピーが短い | 判断に必要な意味を削っていないか |

---

### 4. 見せる情報より、見せない情報を評価する

AI生成UIは情報を盛り込みすぎる傾向がある。
良いUIは、ユーザーが今考えなくてよい情報を下げる。

評価すること:

- 主判断に不要な情報を前面に出していないか
- 詳細・ログ・補足・監査情報は必要時に掘れるか
- 重要情報と装飾が競合していないか
- ユーザーの注意を分散させていないか

---

## 評価カテゴリと配点

合計100点。
ただし後述の「上限ルール」により、特定条件を満たさない場合は総合点に上限を設ける。

| カテゴリ | 配点 |
|---|---:|
| A. ユーザー目的・ユースケースストーリー | 15 |
| B. 問いの順番・Z配置 | 20 |
| C. 意図整合性 | 20 |
| D. 情報構造・認知負担 | 15 |
| E. モーション・マイクロインタラクション | 10 |
| F. トーンマナー・視覚品質 | 10 |
| G. 状態設計・誤判断防止 | 10 |
| **合計** | **100** |

---

# A. ユーザー目的・ユースケースストーリー（15点）

## 評価すること

画面が、抽象的な「管理」「確認」「分析」ではなく、具体的なユーザーの状況・判断・行動から逆算されているか。

## 見るべき問い

- 誰がこの画面を見るのか
- どんな状況でこの画面に来るのか
- その人は何を達成したいのか
- 何を判断しなければならないのか
- 判断を間違えると何が起きるのか
- 何に不安を感じているのか
- 画面を見た後、何をするのか

## 採点

| 点 | 基準 |
|---:|---|
| 0 | 目的がない。情報や部品が並んでいるだけ。 |
| 3 | 「進捗を見る」「タスクを管理する」など表面的な目的はある。 |
| 6 | 操作目的は分かるが、判断目的が弱い。 |
| 9 | ユーザーが何を判断したいかがある程度明確。 |
| 12 | 利用文脈、不安、判断、次行動が具体化されている。 |
| 15 | ユースケースストーリー全体から画面が逆算されている。 |

## 低評価例

> ユーザーがプロジェクトの進捗を確認できる画面。

## 高評価例

> プロジェクト責任者が、週次レビュー前に、遅延リスクの高い作業・原因・影響範囲を把握し、どのチームへ確認・介入すべきかを判断する画面。

---

# B. 問いの順番・Z配置（20点）

## 評価すること

ユーザーが知りたいことが、知りたい順番で配置されているか。
視線の流れが、ユーザーの理解・判断・行動の流れと一致しているか。

## 基本的なZ配置

| 位置 | 役割 |
|---|---|
| 左上 | これは何の画面か。現在の文脈は何か。 |
| 右上 | 対象、期間、権限、信頼性、ステータス。 |
| 中央左 | 最重要情報。今、何を見るべきか。 |
| 中央右 | 理由、根拠、比較、補足。 |
| 左下 | 詳細、履歴、関連情報、監査情報。 |
| 右下 | 次の行動。承認、保存、再実行、問い合わせなど。 |

## 見るべき問い

- 最初に見るべき情報が明確か
- 次に確認したくなる情報が近くにあるか
- 主張と根拠が離れすぎていないか
- CTAは判断材料を見た後の自然な位置にあるか
- 詳細情報は主判断を邪魔せず掘れるか
- 視線は流れているだけでなく、問いの順番になっているか

## 採点

| 点 | 基準 |
|---:|---|
| 0 | 情報がシステム都合で並んでいる。 |
| 4 | 一般的なレイアウトはあるが、問いの順番ではない。 |
| 8 | 重要情報は目立つが、理解順序が弱い。 |
| 12 | 判断順序がある程度見える。 |
| 16 | ユースケースの問いがZ配置に自然に乗っている。 |
| 20 | 画面全体が「状況 → 重要点 → 根拠 → 詳細 → 行動」の流れとして機能している。 |

---

# C. 意図整合性（20点）

## 評価すること

トーン、コピー、配置、色、余白、動き、CTAが、同じ目的を向いているか。

## 見るべき問い

- この画面の中心意図は何か
- トーンはユーザーの状況に合っているか
- コピーは判断に必要な意味を足しているか
- 動きは理解を助けているか
- 余白は優先順位を支えているか
- 色は状態や重要度を正しく伝えているか
- CTAは急かしていないか
- 装飾が目的と矛盾していないか

## 採点

| 点 | 基準 |
|---:|---|
| 0 | 装飾の寄せ集め。意図が分からない。 |
| 4 | 色やフォントは揃っているが、目的との接続が弱い。 |
| 8 | 画面単位の意図はあるが、細部がずれている。 |
| 12 | 主要要素は目的に沿っている。 |
| 16 | トーン、配置、コピー、動き、状態表現がかなり整合している。 |
| 20 | 体験全体が一つの意図として感じられる。 |

## 典型的な不整合

- 安心して判断したい画面なのに、過剰に派手なモーションを使う
- 慎重な承認画面なのに、CTAだけが強すぎる
- 高級感のあるLPなのに、コピーが事務的すぎる
- 緊急対応画面なのに、装飾が多く認知負担が高い
- AIの推奨だけが強く、根拠やリスクが弱い

---

# D. 情報構造・認知負担（15点）

## 評価すること

ユーザーが理解・判断するために必要な情報が、負担を下げる形で整理されているか。

## 見るべき問い

- 重要情報と補助情報が分かれているか
- 数値や状態に「だから何か」があるか
- 不要情報を前面に出していないか
- 詳細情報は折りたたみ、ドリルダウン、タブなどで退避できているか
- 一画面で判断できる粒度になっているか
- 専門用語やログをそのまま出していないか

## 採点

| 点 | 基準 |
|---:|---|
| 0 | 情報過多または情報不足で判断できない。 |
| 3 | 情報は分類されているが、重要度が不明。 |
| 6 | 重要情報は見えるが、補足や詳細が邪魔。 |
| 9 | 要約、詳細、根拠の階層がある。 |
| 12 | 認知負担を下げるために情報が選別されている。 |
| 15 | ユーザーが迷わず理解・判断できる情報構造になっている。 |

---

# E. モーション・マイクロインタラクション（10点）

## 評価すること

動きが装飾ではなく、状態・因果・注意・理解を支えているか。

## 動きの役割

| 種類 | 役割 |
|---|---|
| 操作反応 | 押せた、選べた、変わったことを伝える |
| 状態表現 | 生成中、分析中、保存中、失敗、完了を伝える |
| トランジション | 一覧から詳細、要約から証跡などの関係を伝える |
| 注意誘導 | 次に見るべき場所を示す |
| 空間表現 | レイヤー、奥行き、遷移の意味を伝える |
| ブランド演出 | 期待感、高級感、知性を演出する |

## 採点

| 点 | 基準 |
|---:|---|
| 0 | 動きがない、または邪魔。 |
| 2 | hoverやfadeなど最低限の装飾的な動きのみ。 |
| 4 | 操作反応やローディングはある。 |
| 6 | 状態変化や画面遷移を理解しやすくしている。 |
| 8 | 注意誘導、因果、状態、詳細展開が意味のある動きで支えられている。 |
| 10 | 時間軸が情報設計の一部として機能している。 |

## 減点対象

- 常時動き続ける装飾
- 重要でない要素のアニメーション
- 進捗を示さない「考えている風」演出
- アニメーション待ちで操作が遅く感じる
- prefers-reduced-motion を考慮していない

---

# F. トーンマナー・視覚品質（10点）

## 評価すること

視覚表現がプロダクトの人格、利用文脈、ユーザーの感情に合っているか。

## 見るべき問い

- ブランドとしてどう見られたいかが表現されているか
- ユーザーの感情状態に合っているか
- 余白、色、タイポグラフィ、アイコン、影が過不足なく使われているか
- 一般的なSaaSテンプレートに回収されていないか
- 視覚の美しさが理解や判断を支えているか

## 採点

| 点 | 基準 |
|---:|---|
| 0 | 視覚的に破綻している。 |
| 2 | 最低限整っているが凡庸。 |
| 4 | 一般的な今風UIとして成立している。 |
| 6 | プロダクトの文脈に合った視覚表現がある。 |
| 8 | トーン、余白、階層、色、コピーが高い水準で揃っている。 |
| 10 | 視覚品質がプロダクト固有の価値や人格を強めている。 |

---

# G. 状態設計・誤判断防止（10点）

## 評価すること

ユーザーが誤解・誤操作・誤判断しないように、状態・リスク・結果が設計されているか。

## 見るべき問い

- 通常、処理中、成功、失敗、警告、空状態が設計されているか
- AIの推奨に根拠と不確実性があるか
- ユーザーが判断を間違えると何が起きるか示されているか
- 重要操作の確認や取り消しがあるか
- 行動後に結果が分かるか
- 色だけに依存していないか

## 採点

| 点 | 基準 |
|---:|---|
| 0 | 状態や失敗が考慮されていない。 |
| 2 | loading/error程度はある。 |
| 4 | 基本状態はあるが、判断リスクが弱い。 |
| 6 | 根拠、警告、確認、結果表示がある。 |
| 8 | 誤判断を防ぐ情報と操作導線が設計されている。 |
| 10 | ユーザーが安心して判断・行動・修正できる状態設計になっている。 |

---

# 上限ルール

次の条件に該当する場合、他の項目が優れていても総合点に上限を設ける。

| 条件 | 総合点上限 |
|---|---:|
| ユーザー目的がない | 35点 |
| 問いの順番がない | 40点 |
| 問いの順番が表面的 | 50点 |
| 問いの順番はあるがZ配置が弱い | 65点 |
| 意図整合性が弱い | 70点 |
| 誤判断防止がない重要画面 | 75点 |
| 見た目だけが優れている | 60点 |

特に重要なルール:

> 問いの順番が設計されていないUIは、ユーザーの理解順序を設計していない。したがって、目的駆動UIとしては高評価にしない。

# AI生成UIでよくある失敗

## 1. 汎用SaaS化

ロゴ、サイドバー、カード、グラフ、CTAはあるが、そのプロダクト固有の問いが見えない。

## 2. 美しいが判断できない

視覚的には整っているが、ユーザーが何を判断すべきか分からない。

## 3. 動きが意味を持っていない

マイクロインタラクションは多いが、状態や因果を伝えていない。

## 4. CTAが早すぎる

不安や根拠に答える前に、登録・購入・承認を迫る。

## 5. 根拠が遠い

主張、警告、推奨の根拠が同じ視線範囲にない。

## 6. 詳細が前面に出すぎる

ログ、件数、トークン数、実行履歴などが主判断を邪魔する。

## 7. トーンがずれている

慎重な判断画面なのに派手すぎる、期待感を作るLPなのに堅すぎる。

# レビュー手順

## Step 1: ユースケースストーリーを書く

以下を1〜3文で書く。

```text
誰が、どんな状況でこの画面に来て、何を判断し、何を不安に思い、画面を見た後に何をするのか。
```

## Step 2: ユーザーの問いを順番に列挙する

例:

```text
1. これは何の画面か
2. 今、何が問題か
3. 自分が対応すべきか
4. なぜそう言えるのか
5. 影響範囲はどこか
6. 詳細を見るならどこか
7. 次に何をすればよいか
8. 実行後に何が起きたか
```

## Step 3: 問いをZ配置にマッピングする

| 問い | 配置 | UI要素 |
|---|---|---|
| これは何の画面か | 左上 | タイトル、文脈ラベル |
| 対象は何か | 右上 | 対象、期間、権限、状態 |
| 何が重要か | 中央左 | 要約、警告、主カード |
| なぜそう言えるか | 中央右 | 根拠、比較、証跡 |
| 詳細はどこか | 左下 | ログ、履歴、関連情報 |
| 次に何をするか | 右下 | CTA、選択肢 |

## Step 4: 意図整合を確認する

以下が同じ目的を向いているか確認する。

- トーン
- コピー
- 色
- 余白
- モーション
- CTA
- 状態表現
- 情報の隠し方

## Step 5: 認知負担を削る

以下を確認する。

- 今見なくてよい情報は何か
- 詳細に逃がせる情報は何か
- 意味づけが足りない数値は何か
- 装飾が重要情報と競合していないか

## Step 6: 状態と誤判断を確認する

以下の状態を確認する。

- 初期表示
- 読み込み中
- 処理中
- 成功
- 失敗
- 警告
- 空状態
- 権限不足
- 再実行
- 取り消し

# AIに渡すプロンプトテンプレート

```text
あなたはUIデザイナー兼UXレビュアーです。
以下のUIを、見た目ではなく、ユーザーの問いの順番と意図整合性から設計してください。

まず、画面を作る前に以下を定義してください。

1. ユーザー像
2. 利用文脈
3. ユーザーが達成したいこと
4. ユーザーが判断したいこと
5. ユーザーが不安に思うこと
6. 判断を間違えた場合のリスク
7. 画面を見た後の次行動
8. ユーザーが順番に知りたい問い
9. その問いをZ配置にどう置くか
10. トーン、コピー、色、余白、動き、CTAの意図

その後、UIを設計してください。

制約:
- 情報を網羅表示しない
- 問いの順番を優先する
- CTAは納得後の自然な位置に置く
- 動きは装飾ではなく、状態・因果・注意・理解を伝えるために使う
- 詳細情報は主判断を邪魔しないように退避する
- 誤判断を防ぐ根拠、警告、確認、結果表示を設計する
- 見た目の今風さより、意図の整合性を優先する
```

# AI生成UIレビュー用チェックリスト

## 目的

- [ ] 誰が使う画面か明確か
- [ ] どんな状況で使うか明確か
- [ ] 何を判断する画面か明確か
- [ ] 判断を間違えた場合のリスクが考慮されているか

## 問いの順番

- [ ] ユーザーが最初に知りたいことに答えているか
- [ ] 次に知りたくなることが自然に配置されているか
- [ ] 主張と根拠が近いか
- [ ] 詳細情報は必要時に掘れるか
- [ ] CTAは判断材料の後にあるか

## 意図整合

- [ ] トーンは利用文脈に合っているか
- [ ] コピーは判断を助けているか
- [ ] 色は状態や重要度を正しく伝えているか
- [ ] 余白は優先順位を支えているか
- [ ] 動きは理解を助けているか
- [ ] CTAはユーザー利益と整合しているか

## 認知負担

- [ ] 情報を出しすぎていないか
- [ ] ログや詳細が前面に出すぎていないか
- [ ] 数値に意味づけがあるか
- [ ] 不要な装飾が注意を奪っていないか

## 状態・安全性

- [ ] 処理中状態が分かるか
- [ ] 成功・失敗・警告が分かるか
- [ ] AIの推奨に根拠があるか
- [ ] 重要操作に確認や取り消しがあるか
- [ ] 行動後の結果が分かるか

# 最終判定の目安

| 点数 | 判定 |
|---:|---|
| 0〜34 | 目的駆動UIとして成立していない |
| 35〜49 | 見た目や構成はあるが、問いの順番が弱い |
| 50〜64 | 一部の問いには答えているが、意図整合が不足 |
| 65〜79 | 実用可能。目的、問い、構造、動きが概ね揃っている |
| 80〜89 | 高品質。細部まで意図がかなり整合している |
| 90〜100 | 非常に高品質。体験全体が一つの意図として統合されている |

# まとめ

AI生成UIの評価では、見た目の良さに引っ張られない。
特に、以下を満たさない場合は高評価にしない。

1. ユーザーの目的が具体化されている
2. ユーザーの問いの順番が設計されている
3. 問いが視線順序に配置されている
4. トーン、コピー、余白、色、動き、CTAが同じ意図を向いている
5. 認知負担を下げるために、見せない情報が設計されている
6. 状態・根拠・リスク・結果が分かる

最重要原則:

> 良いUIは、情報を配置するのではなく、ユーザーの問いの順番を配置する。
> 良いデザインは、意図の一貫性を、視覚・言葉・動き・構造に分配する。
