Nagare Workflow State Design

作成日: 2026-05-30 / 対象: ユーザー入力から Work Item 完了までの状態遷移、状態説明、Agent 終了時の記録データ

1. 基本方針

Nagare は、ユーザーの依頼を WorkItem として保存し、各 Agent / action の終了後に必ず dispatch / decision 層へ戻して、次に誰が何をするかを決める。

ターゲット設計: Work の後も Review の後も、直接次 Agent に固定せず dispatch に戻す。dispatch が run_work、run_review、approve、ask_human、handoff を決める。

Review 指摘時: Review が指摘点を返した場合は、指摘を next_notes / ReviewResult.requested_changes として保存し、dispatch が原則 work に戻す。必要なら別 Agent に handoff または再 dispatch する。

Plan 作成: 複雑な作業で必要な Plan は、専用 status ではなく work purpose の Planner Agent が作る。Plan 作成後も dispatch に戻り、Review に回すか、実装 Work Agent に進めるかを決める。

状態の単純化: ready_for_review のような「次工程名を含む status」はターゲット設計では不要。次工程は status ではなく WorkflowDecision / DispatchPlan が表す。

検査/CI: 独立した 独立検査工程は置かない。Review Agent がスキル、CI、テスト、成果物確認を必要に応じて実行し、結果を ReviewResult.findings / requested_changes / criteria_results に残す。

Approval: OK は approve の1アクションで done にする。NG は理由付きの HumanDecision(decision_type=reject) を保存し、既存の dispatch / recovery 候補を古い判断として無効化して ready に戻す。次は dispatch が再判断する。

状態は次アクションそのものではない。 次に何をするかは WorkItemStatus だけでなく、WorkItemSnapshot、最新の ReviewResult、RecoveryPlan、DispatchPlan、HandoffPacket から WorkflowDecision が決める。

2.5 次アクション決定層

状態遷移図は主要フローを示す。実際の次アクションは状態内で固定されず、WorkflowDecision が snapshot 全体から決定する。

状況 WorkflowDecision 理由
ready で Work output があり、ReviewResult がまだない run_review Work Agent の出力を評価する必要がある。
Review pass approve 通常 Item は Item 完了時にユーザーが最終確認する。
複雑な Item で Plan が未作成 run_agent target=planner Plan 作成も Work の一種として扱い、Planner の output を次工程の判断材料にする。
Plan 作成済みで Plan review が必要 run_review 実装に入る前に Plan の妥当性、抜け、リスクを review する。
Plan が承認/採用済み run_agent target=worker Planner の next_notes と Plan output を実装 Agent の入力にする。
Review で CI / テスト / 成果物確認が必要 run_review Review Agent が必要な検査を実行し、指摘があれば requested_changes として返す。
Review request_changes / criteria failed / contract violation create_recovery_plan 同じ Agent で再実行するか、別 Agent に渡すか、再 dispatch するかを RecoveryPlan で決める。
HandoffPacket 作成済み dispatch / accept_dispatch / run_agent 引き継ぎ先を選び直すため dispatch に戻る。
accepted RecoveryPlan がある apply_recovery_plan 復旧案に従って work または review の AgentRun を再作成する。

3. ターゲット WorkItem 状態と意味

Status 意味 主な次アクション 遷移条件
ready dispatch / decision 待ち。作成直後だけでなく、work、review、recovery 適用後もここへ戻る。 dispatch ユーザー作成直後、Agent / action 終了後、HumanFeedback 後、RecoveryPlan 適用後。
agent_running Agent 実行中を表す一時状態。永続 ledger では実行終了後の状態に更新される。 wait UI server などが background 実行中表示に利用する。
awaiting_approval Item の回答/成果が完了し、ユーザーの最終確認だけが残っている。 approve / reject Review pass 後。OK は1アクションで完了し、NG は理由付きで差し戻して dispatch に戻す。
needs_input Agent がユーザーに具体的な質問を持っている。AI作業フィードバック待ちではない。 answer_question AgentOutputRecord.questions が空でない場合。
needs_handoff 別 Agent への引き継ぎが必要。HandoffPacket の作成または dispatch 待ち。 create_handoff / dispatch / run_agent Work Agent の next_action が handoff 系の場合、または明示的に handoff を作成した場合。
needs_recovery Review 指摘、契約違反、CI/test failure などにより復旧判断が必要。 create_recovery_plan / apply_recovery_plan RecoveryPlan により同じ Agent 再実行、Review 再作成、handoff、redispatch を選ぶ。
done ユーザーが完了を承認した最終状態。 done awaiting_approval で approve された場合。

現行実装との差分: 現在の ledger には ready_for_review、changes_requested などがある。ターゲット設計では、次工程名を含む永続 status を減らし、WorkflowDecision、ReviewResult、RecoveryPlan から導出する表示状態に寄せる。

4. 状態遷移ルール

起点 イベント 条件 遷移先 備考
ユーザー入力 Work Item 作成 title / description などを保存 ready workflow_mode と approval_policy は明示値、Domain override、Project workflow default の順に解決して保存する。
ready dispatch / decision 次に必要な actor / action を判断 agent_running / awaiting_approval / needs_input / needs_handoff / needs_recovery WorkflowDecision と必要に応じて DispatchPlan を保存する。
agent_running Work Agent 終了 成功、質問なし、handoff 要求なし ready Work の結果を AgentOutputRecord に保存し、次を dispatch に判断させる。
agent_running Planner Agent 終了 Plan output を返した ready Plan も AgentRunPurpose::Work。AgentOutputRecord.summary に Plan 本文、next_notes に実装 Agent への指示を残す。ファイル作成が指示された場合だけ Artifact に plan.md 等を保存する。
agent_running Work Agent 終了 questions が空でない needs_input next_action=answer_question だけでは入力待ちにしない。
needs_input ユーザー回答 answer が空でない ready HumanFeedback を保存し、次の Agent 実行 prompt に反映する。
ready dispatch / decision Work output があり、ReviewResult がまだない agent_running Review Agent を実行する。
agent_running Review Agent 終了 pass、criteria 充足、必要なCI/テスト確認済み ready ReviewResult を保存し、dispatch が awaiting_approval を選ぶ。
agent_running Review Agent 終了 request_changes、criteria failed、または Review 出力の契約違反 ready ReviewResult / warnings を保存し、dispatch が原則 work 再実行を選ぶ。必要なら needs_recovery で復旧案を作る。
ready dispatch / decision Review 指摘があり、同じ Agent で直せる agent_running Review 指摘を prompt / run packet に入れて Work Agent を再実行する。
ready dispatch / decision Agent 選定からやり直すべき needs_handoff または ready HandoffPacket を作るか、再 dispatch して target agent を選び直す。
needs_handoff HandoffPacket 作成後の継続 別 Agent に渡す必要がある ready handoff dispatch -> accept_dispatch -> run_agent の順に dispatch に戻る。
agent_running Review Agent 終了 pass、criteria 充足、CI / テスト / 成果物確認も問題なし ready ReviewResult を保存し、approval_policy=manual_final_approval なら dispatch が approval を選ぶ。auto_complete_on_review_pass なら WorkflowDecision が done を選び、承認記録を残して完了する。
awaiting_approval 承認 approval_gate.ready=true done ユーザーが OK を1アクションで確定する。
awaiting_approval 差し戻し approval_gate.ready=true かつ理由あり ready HumanDecision(decision_type=reject) を保存し、dispatch が次の作業方針を再判断する。

5. Agent 終了時に残る記録データ

Artifact の定義: Artifact はユーザーが作成を指示したファイル、またはそのファイル作成に必要なファイルだけを指す。実行ログ、raw output、diff、CI/test log は ExecutionRecord として証跡に残す。

Output の定義: AgentOutputRecord.fields.summary が Agent としての生成結果。質問回答タスクでは回答そのものが output になる。

5.1 Agent run 共通

Record 保存タイミング 主なフィールド 用途
AgentRun Agent プロセス終了時 purpose, agent_profile_id, command, status, exit_code, execution_record_id どの Agent が何を実行したかを追跡する。
ExecutionRecord Agent プロセス終了時 record_type=run_log, uri, title stdout/stderr、command、exit code の証跡。
ExecutionRecord Git work tree に差分がある場合 record_type=changed_files / diff_patch 作業差分の証跡。Artifact には数えない。
Evidence Agent run 保存時 claim, basis, execution_record_id, produced_by 実行が成功/失敗した根拠を言語化する。
ResolvedSkillContext Agent run 保存時 agent、context refs、resolved skills Agent 実行前に解決された context を追跡する。
ResolvedRunPacket Agent run 保存時 goal、constraints、review checks、context refs Agent に渡した実行パケットを証跡化する。

5.2 Agent purpose 別

Agent / purpose 追加で保存する record 状態への影響 備考
dispatch_preview DispatchPlan, raw output の ExecutionRecord 原則 status は変えない。accepted plan があれば次に work agent を実行できる。 target agent、summary、risks、missing_information、selection_warnings を保存する。
work AgentOutputRecord, 必要に応じて Artifact 成功なら ready に戻り、dispatch が review / approve / handoff などを決める。質問ありなら needs_input。 summary がユーザー向け出力、next_notes が次 Agent への判断材料。
review AgentOutputRecord, ReviewResult 終了後は ready に戻り、dispatch が approve / work 再実行 / handoff を決める。 criteria、CI、テスト、成果物確認の結果は ReviewResult に保存する。
workflow_supervision WorkflowDecision 直接状態を変えるのではなく、次の workflow action を決める。 deterministic decision の代替として supervisor agent が判断する。

5.3 Plan 作成 Agent の扱い

項目 扱い 保存先 補足
Agent 種別 Work Agent の一種 AgentRun.purpose=work, agent_profile_id=planner など status に planning を増やさず、dispatch が planner を選ぶ。
Plan 本文 Agent としての生成結果 AgentOutputRecord.fields.summary / completed Plan を回答として返すだけなら Artifact ではない。
Plan ファイル ユーザーまたは workflow がファイル作成を要求した場合だけ Artifact Artifact docs/plan.md など。作成指示がない場合は Artifact を作らない。
次 Agent への指示 判断材料 AgentOutputRecord.fields.next_notes 実装順、未決事項、リスク、検証観点を次の dispatch / Agent に渡す。
Plan review 通常の Review ReviewResult 複雑な作業では実装前に Plan の妥当性を review できる。
Plan 後の遷移 dispatch に戻る WorkflowDecision / DispatchPlan 次に Plan review、実装 Work、追加質問、handoff のどれに進むかを dispatch が決める。

5.4 ドメイン専用 Planner / Worker / Reviewer

Agent Profile は domain と role の組み合わせで分化できる。状態遷移は増やさず、dispatch が候補 Agent Profile の role、specialties、description、working_dir を見て選ぶ。

例 role specialties 用途
code-planner planner code, architecture, risk コード変更の手順、影響範囲、検証観点を計画する。
code-worker worker rust, tests, refactor Plan または Review 指摘に従って実装する。
code-reviewer reviewer correctness, regression, testability 実装や Plan をレビューし、修正点を返す。
docs-planner planner documentation, information-architecture 文書構成、読者、章立てを計画する。
research-worker worker research, synthesis, sources 調査、根拠整理、要約を担当する。
ui-reviewer reviewer ui, accessibility, workflow 画面仕様、導線、表示文言をレビューする。

実装上は AgentRunPurpose を増やさない。Plan 作成と実装はいずれも work、レビューは review とし、agent_profile_id と role で担当を区別する。

dispatch prompt には候補 Agent Profile の role を含める。これにより code-planner、docs-worker、research-reviewer のような domain × role の選択ができる。

Artifact Type: ドメイン内の成果物固有の判断基準は Agent ではなく Artifact Type に置く。.nagare/artifact-types/<id>.toml が domain_id、description、artifact_types、rubric、dispatch_hints を持ち、Planner / Worker / Reviewer は同じ Artifact Type Rubric を目的に応じて使う。関係図は Domain / Artifact Type / Rubric / Agent Design を参照する。

Workflow Policy: Project は .nagare/project.toml の [workflow] に default_progress_mode と approval_policy を持つ。Artifact Type は workflow.progress_mode と workflow.approval_policy を任意で持ち、新規 Work Item 作成時に Project default を上書きする。

Settings では Work / Review / Dispatch / Supervisor の固定 default 割り当てを主機能として出さない。ユーザーは Agent Profile の role、specialties、description、working_dir を整備し、dispatch がそれらを見て都度選ぶ。fallback 設定は内部・高度設定として扱う。

5.5 Agent 以外の終了イベント

イベント 保存する record 状態への影響 用途
ユーザー回答 HumanFeedback needs_input から ready Agent の質問への回答。AI作業評価とは別。
Handoff 作成 HandoffPacket needs_handoff open questions、artifact ids、execution record ids、review ids を含める。
Recovery 作成/承認/適用 RecoveryPlan, 適用時は新しい AgentRun 復旧後は通常の work/review flow へ戻る。 failure_class に review_changes, contract_violation, missing_artifact などを保存する。
承認 HumanDecision done Item 完了時のユーザー確認。ここで人間が最終判断する。

7. 画面表示での読み替え

内部状態 UI 表示 表示理由
ready + WorkflowDecision=run_review レビュー実行待ち Work Agent は終わったが ReviewResult がまだない。
awaiting_approval または ready + WorkflowDecision=approve 完了確認待ち / 承認待ち Review pass 済み。OK はワンアクションで完了でき、NG は理由付きで再 dispatch する。
ready + HumanDecision=reject 差し戻し / 再dispatch待ち 理由付き reject が保存され、既存の dispatch / recovery 候補は supersede されている。
ready + Review 指摘あり 修正作業待ち dispatch が Review 指摘を次の Work Agent 入力へ渡す。
needs_recovery 復旧判断待ち 同じ Agent 再実行では足りない可能性があり、RecoveryPlan / handoff / redispatch を選ぶ。
needs_input 回答が必要 AgentOutputRecord に具体的な questions がある。
AI作業フィードバック収集 ON AIへのフィードバック Item 完了確認とは別の任意収集。正しかった/正しくなかったとコメントを保存する想定。
