Nagare Domain / Rubric / Agent Design

Last updated: 2026-05-31 / Scope: Nagare project design / Related: Workflow State Design, spec.md, design.md

この文書は Domain Group、Domain Profile、Rubric、Agent Profile、Workflow Policy の関係を定義する。状態遷移そのものではなく、dispatch と review が何を判断材料にするかを扱う。

Work Item Domain Group Domain Profile Rubric Agent Profile Workflow / Dispatch

1. 基本方針

Domain Group はターゲット設計に含める。 現行実装はDomain中心でも、設計上は最初から Domain Group -> Domain の2階層を正とする。後からGroupを足す前提にすると、Work Item、dispatch context、Rubric改善履歴の置き場所が揺れるため、データモデルとSettings UIはGroup込みで設計する。

Domain は成果物や判断領域を表す。 Agent の種類ではない。たとえば frontend-ui、rust-cli、docs のように、作成物と評価基準がまとまる単位として扱う。

Domain Group は大きな業務領域を表す。 たとえば software-development の下に frontend-ui、backend-api、database-migration、developer-docs を置く。Group は共通知識、共通dispatch_hints、共通workflow defaultを持ち、Domain は生成物ごとのRubricで具体化する。

Rubric は Domain に属する。 Planner / Worker / Reviewer のどの Agent も、同じ Domain Rubric を目的に応じて使う。Rubric を Agent 個別に閉じ込めないことで、作成基準と評価基準のズレを減らす。

Agent は実行能力と役割を表す。 Agent Profile は role、specialties、description、working_dir を持ち、dispatch が Work Item と Domain を見て実行Agentを選ぶ。

2. 階層と解決順序

Domain Group は分類と共通知識の層、Domain は生成物別の具体基準の層として扱う。Work Item は作成時に Domain Group / Domain / workflow policy を解決し、その結果を ledger に固定する。

1. Work Item明示値 作成時に指定された domain_id、workflow_mode、approval_policy を最優先する。
2. Domain override 成果物固有のRubric、dispatch_hints、workflow override を適用する。
3. Domain Group default 共通知識、共通Rubric、共通workflow default を補完する。
4. Project default Domain未指定や未設定項目の最後の既定値として [workflow] を使う。
項目 解決順序 保存先 意図
進行方針 Item明示値 -> Domain override -> Domain Group default -> Project default WorkItem.workflow_mode 作成後にDomain設定が変わっても、実行中Itemの挙動を変えない。
最終承認方針 Item明示値 -> Domain override -> Domain Group default -> Project default WorkItem.approval_policy 低リスクDomainは自動完了、高リスクDomainは人間承認待ちにできる。
Rubric Domain Rubric + Domain Group common Rubric + Work Item acceptance criteria Review prompt / ReviewResult 共通基準と成果物固有基準を同時にReviewへ渡す。
Agent選択 Work Item context + Domain / Group hints + Agent Profile候補 DispatchPlan Settingsの固定defaultではなく、dispatchが目的別に選ぶ。

5. Domain Group の粒度

Domain Group は、知識体系や前提が大きく共有される単位で切る。ソフトウェア開発のように生成物が多い領域では、Groupを software-development とし、その下に成果物別Domainを置く。

Domain Group Domain例 共有する知識 Domain側で分ける知識
software-development frontend-ui, backend-api, database-migration, developer-docs, test-automation 変更管理、互換性、レビュー観点、CI、セキュリティ、既存設計の尊重。 UIの視認性、API契約、DB移行安全性、ドキュメント読者、テスト信頼性。
business-operations sales-report, support-analysis, finance-summary データ出所、数値整合性、意思決定に必要な要約。 レポート形式、分析粒度、監査性、用語定義。

6. 実行時の使われ方

作成時

  • ユーザーがDomain Group / Domainを選ぶ、または未指定にする。
  • 未指定時はProject workflow defaultを使う。
  • Domain指定時はDomain override、Group default、Project defaultの順に解決する。

Dispatch時

  • Agent Profileのroleとspecialtiesを候補評価に使う。
  • Domain Groupの共通知識とDomainのdispatch_hints / rubricを文脈として渡す。
  • Work / Review の固定default UIには戻さない。

Review時

  • Reviewはacceptance criteria、Domain Groupの共通基準、Domain Rubricで評価する。
  • CI、テスト、artifact確認もReview内の証跡として扱う。
  • pass後はapproval policyで完了方法が分岐する。

7. 設定データ例

Domain Group

大きな業務領域の共通知識を置く。下位Domainに共通する前提だけを保持する。

[domain_group]
id = "software-development"
display_name = "Software Development"
description = "Code, UI, API, docs, tests."
shared_knowledge = [
  "Prefer small reviewable changes.",
  "Preserve existing public contracts."
]
common_rubric = [
  "Change is traceable to requirements.",
  "Relevant tests or review evidence exist."
]
dispatch_hints = [
  "Select agents by role and working_dir."
]

[domain_group.workflow]
default_progress_mode = "finish_first"
approval_policy = "manual_final_approval"

Domain

生成物別の具体的なRubricを置く。Groupの共通知識を継承して具体化する。

[domain_profile]
id = "frontend-ui"
group_id = "software-development"
display_name = "Frontend UI"
artifact_types = ["html", "css", "screenshot"]
rubric = [
  "Primary workflow is visible.",
  "Text does not overlap.",
  "Controls are reachable on mobile."
]
dispatch_hints = [
  "Prefer UI-capable worker/reviewer."
]

[domain_profile.workflow]
approval_policy = "manual_final_approval"

Work Item

作成時に解決済み値を保存する。後からGroupやDomainが更新されても暗黙には変えない。

{
  "id": "work_0001",
  "domain_group_id": "software-development",
  "domain_id": "frontend-ui",
  "workflow_mode": "finish_first",
  "approval_policy": "manual_final_approval",
  "acceptance_criteria": [
    "SettingsでDomainが選べる"
  ]
}

8. Settings UI 方針

画面 扱うもの 主操作 表示しないもの
Workflow Project default default_progress_mode と approval_policy を保存する。 Work / Review agent の固定default。
Domains 大きな業務領域と生成物別の知識 1つのタブ内に Domain Groups と Domains の2リストを表示する。各リストの直上に新規作成ボタンを置き、各行から編集・削除する。 Agent runtimeやadapterの設定。Domain Group と Domain を別タブに分けた長い管理導線。
Agents 実行能力と役割 role、specialties、description、working_dir、adapter、担当Domain Group / Domainを編集する。Project-local Agent は削除できる。 Domain Rubricそのもの。

追加ストーリー。 ユーザーは Settings で Domain Group を1つ作成し、その配下に複数の Domain を作る。Domain は Group を選ぶだけで共通知識と workflow default を継承できる。Agent を追加するときは、担当できる Domain Group と Domain を選び、dispatch はその scope を候補評価に使う。

最小操作。 Group 作成、Domain 作成、Agent 作成はそれぞれ1フォームで完結する。Work Item 作成では Domain を選べば Group はDomainの所属から解決され、Groupだけを選ぶ作業も許可する。

8A. UI評価

観点 評価 理由
状態の可視性 Pass Settings は Workflow、Domains、Agents の順に表示し、Domainsタブ内の2リストでDomain GroupとDomainの件数、所属、workflow、source、編集・削除操作を確認できる。
現実の作業語彙 Pass 大分類は Domain Group、成果物別基準は Domain、実行主体は Agent として分け、RubricをAgent名に埋め込まない。
エラー予防 Pass Domain と矛盾する Domain Group 指定は Work Item 作成時に拒否する。存在しない Group / Domain をAgent scopeに指定することも拒否する。
効率 Pass OK系はフォーム送信1回で保存し、Agent削除も対象確認後に1操作で完了する。既定Agentの固定Default変更UIは置かない。
最小性 Pass Domain Groupに共通基準、Domainに成果物別基準、Agentに実行能力を置くため、設定項目の重複が少ない。

9. Prompt / Rubric 改善の将来設計

対象 改善に使う記録 改善先 注意点
Domain Group 複数Domainに共通する失敗傾向、用語の揺れ、共通前提、横断的なReview指摘 .nagare/domain-groups/<id>.toml の shared_knowledge / common_rubric Groupに置くのは横断知識だけにする。成果物固有の細かい基準はDomainへ下ろす。
Domain Rubric ReviewResult.criteria_results, findings, requested_changes, approval / reject rationale .nagare/domains/<id>.toml の rubric 個別失敗をそのまま追加せず、繰り返し使える評価基準に抽象化する。
Agent Profile AgentRunの成功/失敗、出力契約違反、質問発生、RecoveryPlan description, specialties, output contract Domain固有の品質基準はAgentではなくDomainへ置く。
Dispatch hints DispatchPlanの選択、fallback、selection_warnings、実行後のReview結果 DomainProfile.dispatch_hints 特定Agent名への固定ではなく、候補選択の条件として残す。
Workflow policy 承認待ちの頻度、reject理由、自動完了で問題が出たDomain / Domain Group Project [workflow]、Domain Group workflow default、Domain workflow override GroupとDomainでリスクが違うため、Project defaultだけで全てを決めない。

10. 実装ステータス

対象 状態 現時点の扱い 次に実装する場合の焦点
Project Workflow 実装済み .nagare/project.toml の [workflow] と Settings UI で扱う。 Domain Group default、Domain override、Work Item明示指定の解決順序で統合済み。
Domain Profile 実装済み .nagare/domains/<id>.toml、Settings UI、Item作成時のDomain選択で扱う。group_id を保存できる。 Rubric改善履歴の管理。
Domain Group 実装済み .nagare/domain-groups/<id>.toml、Settings UI、dispatch prompt、Agent scope、WorkItem作成時のGroup固定保存で扱う。 Group改善履歴の管理。
Agent Selection 実装済み Agent Profile、Domain Group、Domain Profileがdispatch材料になる。 実行記録からAgent Profile / Rubricの改善候補を作る。

11. 設計上の禁止事項

  • RubricをAgentごとの隠れたプロンプトに閉じ込めない。Domainに置き、作成と評価で共有する。
  • Domain Groupに成果物固有の細かいRubricを置きすぎない。Groupは共通知識、Domainは生成物別の具体基準に分ける。
  • Work / Review / Planner の固定default選択をSettingsの主機能に戻さない。Agent選択はdispatchがAgent ProfileとDomainを見て判断する。
  • 実行ログ、raw output、CIログ、diffをArtifactと呼ばない。証跡は ExecutionRecord、ユーザー指示の作成物は Artifact として分ける。
  • Domain Group / Domain overrideを既存Work Itemへ暗黙適用しない。Work Itemは作成時点の解決済み方針を持つ。
